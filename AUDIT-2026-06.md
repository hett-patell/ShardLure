## Findings (all VERIFIED against source unless noted)

### HIGH-1 — Oversized Cowrie log line permanently stalls ingest (attacker DoS)
`internal/ingest/cowrie/ingest.go:387-389` (2 MiB scanner cap) + `:104-106` (early
error return before `SetIngestState` at `:129`). A single >2 MiB JSON line (attacker
controls `input`/`username`/`password`) makes `parseReader` return `bufio.ErrTooLong`;
`IngestFileAppend` returns before persisting the offset, so every 5s tick re-reads the
same poison line forever. Cowrie ingest is silently, permanently blind from that offset.
Fix: on ErrTooLong, skip past the next newline (advance offset) instead of erroring;
bound stored field lengths (see MED-2). VERIFIED at source.

### HIGH-2 — Dashboard unreachable when SHARDLURE_DASH_TOKEN is set (availability regression)
`internal/web/server.go:318-336` reads the token header-only (correctly, for security),
but the page routes themselves are gated: `handleIndex` (`server.go:311`) and
`handleIntelPage` (`intel.go:92`). A browser navigating to `/intel?token=X` sends no
Authorization header, so the HTML 401s before the client bootstrap JS (which would
copy ?token= into sessionStorage and set the header on /api/ calls) ever runs. Net:
the documented token feature makes the whole dashboard unreachable via browser.
Fix: accept `?token=` as a fallback on the PAGE routes only (handleIndex/handleIntelPage),
keep API routes header-only; or serve a tiny bootstrap page that sets the header then
redirects. VERIFIED at source (server crypto is constant-time and correct).

### MED-1 — Partial trailing line consumed by offset → silent event loss
`internal/ingest/cowrie/ingest.go:390-396, 113-116`. bufio.Scanner returns the final
line even without a trailing newline; `consumed += len+1` then clamps to file size,
advancing the offset past an incomplete last line that gets dropped when completed.
Fix: only advance offset over newline-terminated lines. VERIFIED.

### MED-2 — No per-field size bound on stored event fields (unbounded growth; amplifies HIGH-1)
`internal/ingest/cowrie/ingest.go:466-481`. Raw/Command/Username/Password stored up to
the 2 MiB cap. Fix: truncate to e.g. 8-64 KiB in toEvent. VERIFIED.

### MED-3 — NAT64 prefix bypasses SSRF guard
`internal/capture/fetch.go:174-199` + `reservedRanges :164`. `64:ff9b::/96` is not in
the block list; `64:ff9b::a00:1` (→10.0.0.1) and `64:ff9b::7f00:1` (→127.0.0.1) return
IsPrivate=false/IsLoopback=false/IsGlobalUnicast=true (reproduced empirically). On a
NAT64-enabled host, an attacker-harvested URL `http://[64:ff9b::7f00:1]/` reaches an
internal host. Conditional on NAT64 presence → MED. Fix: add 64:ff9b::/96 (and
2002::/16 6to4) to reservedRanges, or re-check embedded v4. VERIFIED empirically.

### MED-4 — Enrich HTTP client has no SSRF guard (latent; safe only via one caller's validation)
`internal/intel/enrich/enrich.go:88` uses a default client (no safeDial, follows
redirects anywhere). Safe TODAY only because the sole caller `handleIntelEnrich`
(`api_intel.go:799-813`) validates the IP to global-unicast first. Several providers
(shodan/otx/abuseipdb) raw-concatenate the IP without url.PathEscape. Any future caller
inherits an unguarded client. Fix: give enrich the same safe transport + PathEscape all
providers + assert net.ParseIP at LookupAll entry. VERIFIED (architectural).

### MED-5 — Dashboard auth default-OFF; paid-API + export reachable if bound to 0.0.0.0
`internal/web/server.go:319` (empty token → open). With no token and a non-loopback
bind, `/api/intel/enrich` (paid quota), bazaar upload (abuse.ch key), and credential
wordlist export are unauthenticated. Fix: fail-closed — refuse non-loopback bind when
token unset, or always gate paid/export routes. VERIFIED.

### LOW-1 — copytruncate / same-inode regrowth can skip events (theoretical)
`internal/ingest/cowrie/ingest.go:93-99`. Truncation detected only by size<offset;
same-inode truncate-then-regrow past old offset skips bytes. Standard logrotate+create
is handled. Fix: content fingerprint alongside inode+offset.

### INFO — Toolchain stdlib vulns (NOT a code defect)
govulncheck: 4 reachable vulns, ALL in the go1.26.2 standard library (net/http HTTP/2),
fixed in go1.26.3. Dependency/module vulns found are NOT reachable. Action: build
releases with go1.26.3+. `golang.org/x/text v0.3.8` flagged but not reachable.

attacker data shipped as JSON, escaped client-side via esc/escHtml/safeUrl; raw
